# Function Chain API for Search Rerank

**Status:** Proposed

**Date:** 2026-07-15

**Scope:** Milvus Lite support for the PyMilvus Function Chain API on ordinary Search

## 1. Summary

Milvus Lite will support the public Function Chain API added to Milvus and PyMilvus by accepting structured `FunctionChain` protobuf messages on ordinary `SearchRequest` RPCs.

The first release supports one `L2_RERANK` chain with ordered `map`, `sort`, and `limit` operators. Supported expressions are `decay`, `num_combine`, `round_decimal`, and `rerank_model`.

The implementation will compile the public protobuf plan into Milvus Lite's existing `FuncChain`, `DataFrame`, operator, and expression runtime. Function Chain remains a gRPC Adapter capability: the internal `Collection.search()` Python API will not accept protobuf objects or a new function-chain parameter.

## 2. Goals

- Accept `SearchRequest.function_chains` generated by the PyMilvus Function Chain DSL.
- Match the first-release Milvus L2 rerank semantics.
- Reuse the existing Milvus Lite function-chain runtime instead of creating a second executor.
- Support ordered public plans with intermediate variables and literal expression arguments.
- Fetch collection fields required by the chain even when users do not request them as output fields.
- Keep final response projection owned by Search.
- Preserve legacy `function_score`, ranker, hybrid rerank, and ordinary search behavior.
- Keep Milvus Lite importable with older PyMilvus protobuf packages that do not define Function Chain messages.

## 3. Non-Goals

- `L0_RERANK`, `L1_RERANK`, ingestion, preprocessing, or postprocessing chains.
- Function Chains on `HybridSearchRequest`.
- Direct Function Chain support in `Collection.search()`.
- XGBoost expressions or FileResource integration.
- Returning intermediate variables as result entity fields.
- Automatically appending sort, limit, group-by, round-decimal, or projection operators.
- Replacing or internally translating the public Chain API into legacy `FunctionScore`.
- Implementing a general user-defined expression language.
- Implementing an optimizer, parallel operator scheduler, or general column-pruning framework.

## 4. User-Facing Behavior

PyMilvus users construct a chain and pass it to ordinary Search:

```python
from pymilvus import FunctionChain, FunctionChainStage
from pymilvus.function_chain import col, fn

chain = (
    FunctionChain(FunctionChainStage.L2_RERANK, name="fresh_popular")
    .map(
        "freshness",
        fn.decay(
            col("published_at"),
            function="exp",
            origin=current_time,
            scale=86400,
        ),
    )
    .map(
        "$score",
        fn.num_combine(
            col("$score"),
            col("freshness"),
            col("popularity"),
            mode="weighted",
            weights=[0.7, 0.2, 0.1],
        ),
    )
    .sort(col("$score"), desc=True, tie_break_col=col("$id"))
    .limit(10)
)

client.search(
    collection_name="articles",
    data=[query_vector],
    anns_field="embedding",
    limit=100,
    output_fields=["title"],
    function_chains=chain,
)
```

The chain executes on the candidates returned by ordinary Search. The final `$score` is serialized through the existing result distance/score field. Collection schema fields fetched only for rerank are returned only when requested through normal `output_fields` projection. Temporary Chain outputs are never returned; an original dynamic field that shares a temporary name keeps its original entity value.

## 5. Alternatives

### 5.1 Compile Public Protobuf Into the Existing Runtime

This is the selected approach.

```text
PyMilvus DSL
  -> FunctionChain protobuf
  -> protobuf decoder
  -> protocol-neutral representation
  -> validation and dependency planning
  -> existing FuncChain runtime
  -> SearchResultData
```

This preserves a clean boundary between the public wire protocol and the execution runtime, supports arbitrary legal operator order, and reuses existing operators and expressions.

### 5.2 Translate Function Chains Into FunctionScore

Rejected because `FunctionScore` cannot naturally represent multiple maps, user-defined intermediate variables, literal expression arguments, or arbitrary sort and limit placement.

### 5.3 Add an Independent Executor to the gRPC Servicer

Rejected because it would duplicate `DataFrame`, expression, and operator behavior and create long-term semantic drift between public Function Chains and legacy rerank execution.

## 6. Architecture

The implementation has four layers.

### 6.1 Protocol Translation

The gRPC translator recursively decodes protobuf messages and produces a protocol-neutral representation. No execution module accepts protobuf objects.

### 6.2 Validation and Planning

The validator checks request-level rules, stage and system-name policies, operator and expression contracts, collection schema inputs, and query-count-dependent requirements. It also computes collection fields that must be fetched internally.

### 6.3 Compilation

The compiler maps the validated representation to the existing `FuncChain` runtime through an explicit expression registry.

### 6.4 Search Integration

The gRPC Adapter executes `Collection.search()` with the planned internal fields, converts hits to a `DataFrame`, executes the compiled chain, then projects results through the original user output field list.

## 7. Module Layout

### 7.1 New Modules

```text
milvus_lite/
├── function/
│   ├── repr.py
│   ├── validator.py
│   ├── compiler.py
│   └── expr/
│       └── num_combine.py
└── adapter/grpc/
    ├── function_chain.py
    └── translators/
        └── function_chain.py
```

#### `milvus_lite/function/repr.py`

Defines immutable, protocol-neutral representations:

```python
@dataclass(frozen=True)
class ColumnArg:
    name: str


@dataclass(frozen=True)
class LiteralArg:
    value: object


@dataclass(frozen=True)
class ExprRepr:
    name: str
    args: tuple[ColumnArg | LiteralArg, ...]
    params: dict[str, object]


@dataclass(frozen=True)
class OpRepr:
    op: str
    expr: ExprRepr | None
    inputs: tuple[str, ...]
    outputs: tuple[str, ...]
    params: dict[str, object]
    read_names: tuple[str, ...]
    write_names: tuple[str, ...]


@dataclass(frozen=True)
class ChainInfo:
    required_inputs: tuple[str, ...]
    written_names: tuple[str, ...]


@dataclass(frozen=True)
class ChainRepr:
    name: str
    stage: str
    ops: tuple[OpRepr, ...]
    info: ChainInfo
```

This module must not import protobuf, gRPC, collection schema, or provider implementations.

#### `milvus_lite/adapter/grpc/translators/function_chain.py`

Converts `schema_pb2.FunctionChain` into `ChainRepr`. It is responsible for protobuf structural validity and error context such as `chain[0].op[2].expr.args[1]`.

#### `milvus_lite/function/validator.py`

Validates runtime semantics and collection-schema dependencies. It returns:

```python
@dataclass(frozen=True)
class ValidatedChain:
    repr: ChainRepr
    required_schema_fields: tuple[str, ...]
```

#### `milvus_lite/function/compiler.py`

Compiles `ValidatedChain` into `FuncChain` through explicit operator and expression factories. It does not inspect protobuf or collection schema.

#### `milvus_lite/adapter/grpc/function_chain.py`

Provides thin Search-specific orchestration:

```python
@dataclass(frozen=True)
class SearchFunctionChainPlan:
    chain: FuncChain
    required_fields: tuple[str, ...]
    requested_output_fields: tuple[str, ...] | None


def prepare_search_function_chain(...) -> SearchFunctionChainPlan | None:
    ...


def execute_search_function_chain(...) -> list[list[dict]]:
    ...
```

This keeps protocol decoding and DataFrame conversion out of the already large gRPC servicer.

### 7.2 Modified Modules

- `milvus_lite/function/ops/map_op.py`: add column and literal input bindings.
- `milvus_lite/function/ops/sort_op.py`: support an optional tie-break column.
- `milvus_lite/function/expr/score_combine.py`: reuse the numeric-combine implementation without breaking the legacy class name.
- `milvus_lite/adapter/grpc/translators/search.py`: expose request Function Chains without requiring them to exist in older protobuf packages.
- `milvus_lite/adapter/grpc/servicer.py`: call Search Function Chain preparation and execution helpers.

## 8. Protobuf Decoding

### 8.1 Parameter Values

`FunctionParamValue` is recursively converted to Python values:

| Protobuf field | Python value |
|---|---|
| `bool_value` | `bool` |
| `int64_value` | `int` |
| `double_value` | `float` |
| `string_value` | `str` |
| `bytes_value` | `bytes` |
| `array_value` | `list` |
| `object_value` | `dict[str, object]` |

An unset oneof or an unsupported value kind is rejected. Object keys must be non-empty strings.

### 8.2 Expression Arguments

Each `FunctionChainExprArg` must contain exactly one supported oneof value:

- `column` becomes `ColumnArg`.
- `literal` becomes `LiteralArg` after recursive parameter decoding.

Column names must be non-empty.

### 8.3 Operator Read and Write Names

For `map`, read names are column references in `expr.args`, and write names come from `op.outputs`.

For non-expression operators, read names come from `op.inputs`, and write names come from `op.outputs`.

Literal arguments are not dependencies.

## 9. Request Validation

### 9.1 Request-Level Rules

- Empty `function_chains` preserves existing Search behavior.
- Ordinary Search accepts at most one chain.
- The chain stage must be `FunctionChainStageL2Rerank`.
- A chain must contain at least one operator.
- `function_score` and `function_chains` are mutually exclusive.
- Hybrid Search rejects any non-empty `function_chains` value.
- Unsupported or unspecified stages return an illegal-argument error.

The rejection message for the legacy API conflict should match Milvus:

```text
function_score and function_chains cannot be used together
```

### 9.2 System Names

`$score` and `$id` are runtime virtual columns rather than collection fields.

The collection primary field's real schema name is a read-only alias of `$id` only when it is not a reserved system name. A primary field named `$id` or `$score` is rejected during Function Chain validation before ANN Search. A valid alias has the primary field's INT64 or VARCHAR type, is populated from the search hit ID at runtime, and is not added to `required_schema_fields` or requested as hidden entity output.

| Name | Read | Write |
|---|---:|---:|
| `$score` | yes | yes |
| `$id` | yes | no |
| other `$...` | no | no |

User-created temporary fields must not start with `$`.

### 9.3 Collection Inputs

A name read before any previous operator writes it is a required input.

Required non-system names must resolve to collection fields of one of these types:

- Bool
- Int8, Int16, Int32, Int64
- Float, Double
- VARCHAR
- Timestamptz

The first release rejects vector fields, JSON, Array, Geometry, and dynamic-field subpaths.

To avoid result and schema ambiguity, a public Chain may write `$score` or a non-schema temporary name, but it may not overwrite a collection schema field.

### 9.4 Operator Contracts

Only `map`, `sort`, and `limit` are supported.

`map` requires:

- an expression;
- exactly one output;
- no separate `op.inputs` dependency source.

`sort` requires:

- no expression;
- a non-empty `column` parameter;
- `desc` to be a boolean;
- the first `op.inputs` value to equal `column`;
- an optional `tie_break_col` matching the second input;
- no outputs.

`limit` requires:

- no expression;
- no inputs or outputs;
- a positive integer `limit`;
- a non-negative integer `offset`, defaulting to zero.

The first release does not require sort to be last, limit to be last, or either operator to appear at most once. Operators execute in request order.

## 10. Dependency Planning

Dependency analysis scans operators from first to last.

```text
available = {$id, $score}
required_inputs = []

for each operator:
    for each read name:
        if name is not available:
            add name to required_inputs once
    add each write name to available
```

Example:

```text
map freshness <- decay(published_at)
map $score <- num_combine($score, freshness, popularity)
sort $score, tie-break $id
```

Produces:

```text
required runtime inputs: $score, $id
required schema fields: published_at, popularity
temporary fields: freshness
```

The Search Adapter merges `required_schema_fields` with the user's requested output fields for the internal `Collection.search()` call. It retains the original user output list separately for response projection.

If the current Search behavior uses `None` to represent all output fields, the internal field merge preserves that representation rather than narrowing it.

## 11. Runtime Input Bindings

The public protobuf supports literal expression arguments, while the current `MapOp` accepts only column names.

`MapOp` will support explicit input bindings:

```python
@dataclass(frozen=True)
class ColumnBinding:
    name: str


@dataclass(frozen=True)
class LiteralBinding:
    value: object
```

For each DataFrame chunk:

- `ColumnBinding` reads an existing DataFrame column.
- `LiteralBinding` produces a list containing the literal once per row.

Existing callers that pass string column names continue to work. They are normalized to `ColumnBinding` during `MapOp` construction.

## 12. Expression Compilation and Semantics

The compiler uses an explicit registry:

```python
_EXPR_BUILDERS = {
    "decay": build_decay_expr,
    "num_combine": build_num_combine_expr,
    "round_decimal": build_round_decimal_expr,
    "rerank_model": build_rerank_model_expr,
}
```

Unknown functions are rejected. The implementation must not use `eval`, dynamic class lookup, or unbounded import-by-name behavior.

### 12.1 `num_combine`

Supported modes are `multiply`, `sum`, `max`, `min`, `avg`, and `weighted`.

Rules:

- At least two arguments are required.
- Arguments may be numeric columns or numeric literals.
- `weighted` requires one numeric weight per argument.
- Other modes reject a `weights` parameter.
- Boolean values are not numeric inputs or weights.
- A row containing `None` in any input produces `0.0`, preserving the existing score-combine null behavior.

`NumCombineExpr` is the public expression implementation. The legacy `ScoreCombineExpr` remains available and may delegate to it for its existing non-weighted modes.

### 12.2 `decay`

`decay` requires one numeric or Timestamptz argument and these parameters:

- `function`: `gauss`, `exp`, or `linear`;
- numeric `origin`;
- positive numeric `scale`;
- non-negative numeric `offset`, default zero;
- numeric `decay` strictly between zero and one, default `0.5`.

Timestamptz values are normalized to Unix time using the same timezone interpretation already used by Milvus Lite field translation. A `None` input produces `0.0`.

### 12.3 `round_decimal`

`round_decimal` requires one numeric argument and an integer `decimal` in `[0, 6]`. A `None` input remains `None`.

### 12.4 `rerank_model`

`rerank_model` requires:

- exactly one VARCHAR input;
- a non-empty string array `queries`;
- one query per search query chunk;
- provider parameters supported by the existing rerank provider factory.

Public Function Chain requests may not provide `base_url` or any equivalent case variant. Provider endpoints are trusted server configuration and must never be selected by a Search request. The provider factory may continue to accept server-supplied endpoints in other trusted internal paths.

The query count is checked before ANN Search execution because `nq` is known after placeholder decoding.

The first release supports the providers already implemented by Milvus Lite, currently Cohere. Public Cohere requests may provide only `provider`, `queries`, and `model_name`; unknown provider parameters are rejected before ANN Search. Credentials such as `api_key` are rejected in public Chain parameters and must be resolved from the server process environment. Provider construction and network failures fail the entire search; no partial result is returned.

## 13. Sort Semantics

Public Function Chain sort has an optional tie-break column. The existing `SortOp` always defaults to `$id`, so it must be generalized:

```python
SortOp(
    column: str,
    desc: bool = True,
    tie_break_col: str | None = None,
)
```

- Without a tie-break column, Python stable sorting preserves candidate input order for equal values.
- With a tie-break column, equal primary values are ordered by the tie-break column ascending.
- `None` primary values remain last.
- Legacy builders explicitly pass `$id` and preserve their existing deterministic behavior.

## 14. Search Execution

### 14.1 Preparation

`MilvusServicer.Search()` performs these operations before engine search:

1. Decode the ordinary Search request.
2. Read Function Chains through compatibility-safe access.
3. Validate request-level API conflicts.
4. Convert the public chain to `ChainRepr`.
5. Analyze dependencies against the collection schema.
6. Validate expression parameters and query count.
7. Compile the validated chain.
8. Merge internal required fields into the engine output field list.

Static validation and provider construction should finish before executing ANN Search.

### 14.2 Candidate Search

Public Function Chains do not use the legacy FunctionScore candidate-expansion rules.

- The ANN candidate budget remains the Search request `top_k`.
- The implementation does not apply the existing `top_k * 10` legacy rerank expansion.
- Partition selection, scalar filtering, range search, group-by, and existing Search offset behavior produce the candidate result first.
- A chain `limit` can reduce the candidate count but cannot retrieve candidates absent from the Search result.

### 14.3 DataFrame Construction

Each search hit becomes:

```python
{
    "$id": hit["id"],
    "$score": _hit_score_for_chain(hit, metric_type),
    **hit.get("entity", {}),
}
```

Each query is a separate DataFrame chunk. Empty chunks remain valid.

### 14.4 Chain Execution

The compiled chain executes exactly as sent:

- no implicit map;
- no implicit sort;
- no implicit limit;
- no implicit group-by;
- no implicit round-decimal;
- no implicit select.

If a chain rewrites `$score` without sorting, result order remains unchanged. If a chain omits limit, it returns at most the Search candidate count. Multiple sort or limit operators execute in request order.

### 14.5 Result Conversion

The final `$score` becomes the hit `distance` value consumed by the existing result translator.

Entity output preserves the original Search projection. Collection schema fields are read from the final Chain row, while dynamic fields are restored from the original candidate entity by `$id` within each query chunk. Temporary Chain outputs are never returned. If a temporary output has the same name as an original dynamic field, the response returns the original dynamic value rather than the temporary value.

The response continues to use the original requested output fields and existing ID and score serialization.

## 15. Legacy Isolation

The Search Adapter keeps explicit execution branches:

```python
if public_function_chain is not None:
    execute_public_function_chain()
elif legacy_function_score is not None:
    execute_legacy_function_score()
else:
    execute_plain_search()
```

The public branch must not call `build_hybrid_function_score_chain()`.

Legacy behavior remains unchanged, including:

- implicit merge behavior;
- candidate expansion;
- implicit sort and limit tail;
- legacy group-by integration;
- explicit `$id` tie-breaking;
- legacy provider parameter behavior.

## 16. Error Handling

Invalid public chains raise `SchemaValidationError`, which already maps to the gRPC illegal-argument status.

Errors should include structural context without echoing sensitive values:

```text
function chain[0] op[1] expression num_combine: weighted mode requires 3 weights, got 2
```

Provider responses, document text, query text, credentials, and full expression parameter maps must not be included in logs or validation errors.

Runtime provider and network failures follow the existing Search failure path and fail the complete request.

## 17. PyMilvus and Protobuf Compatibility

Milvus Lite does not add PyMilvus as a project dependency because PyMilvus and Milvus Lite already have an integration relationship that would otherwise create dependency-management problems.

The running gRPC server obtains protobuf classes from the installed PyMilvus package. Compatibility rules are:

- New PyMilvus packages expose `schema_pb2.FunctionChain` and `SearchRequest.function_chains`; the feature is available.
- Old PyMilvus packages omit these definitions; existing Search remains available.
- Module import must not directly resolve missing Function Chain classes or enum constants.
- Request access uses `getattr(request, "function_chains", ())` or descriptor checks.
- Protocol-independent unit tests run without new protobuf definitions.
- Real protobuf and client integration tests run in an environment with a Function-Chain-capable PyMilvus build.

No `pyproject.toml` dependency change is required.

## 18. Performance

The first implementation uses the existing Python DataFrame and per-column expression loops.

The primary controls are:

- fetch only collection fields discovered by dependency planning;
- limit DataFrame size to ordinary Search candidates;
- compile one chain per request;
- avoid copying protobuf objects into the runtime;
- avoid returning hidden input or temporary fields.

The first release does not add chain caching. Provider instances, query lists, and request parameters make request-level caching error-prone and unnecessary for the expected candidate sizes.

## 19. Observability

Debug-level instrumentation may record:

- chain name;
- stage;
- operator count;
- required field names;
- candidate count per request;
- compilation elapsed time;
- execution elapsed time.

Instrumentation must not record:

- query strings;
- document contents;
- full provider parameters;
- credentials;
- literal byte values.

## 21. Implementation Sequence

1. Add the protocol-neutral representation and dependency metadata.
2. Add protobuf parameter and Chain translation.
3. Add schema-aware validation and required-field planning.
4. Generalize `MapOp` for literal bindings.
5. Add `NumCombineExpr` and preserve `ScoreCombineExpr` compatibility.
6. Generalize `SortOp` for an optional tie-break column.
7. Add the expression registry and Repr compiler.
8. Add Search Function Chain preparation and result conversion helpers.
9. Integrate the public branch into ordinary Search.
10. Add real PyMilvus integration tests and run legacy rerank regressions.

## 22. Acceptance Criteria

The design is complete when all of the following are true:

- A Function-Chain-capable PyMilvus client can execute an ordinary Search against Milvus Lite with one L2 chain.
- `map`, `sort`, and `limit` execute in request order.
- `decay`, `num_combine`, `round_decimal`, and `rerank_model` follow the documented contracts.
- Chain-required schema fields are fetched internally and are not leaked.
- Intermediate variables are never returned as entity fields.
- The final `$score` is returned through the existing score/distance field.
- Invalid stages, names, dependencies, parameters, and API conflicts return illegal-argument errors.
- Old PyMilvus environments can still import and use Milvus Lite ordinary Search.
- Existing FunctionScore, hybrid rerank, and plain Search tests remain unchanged and pass.
